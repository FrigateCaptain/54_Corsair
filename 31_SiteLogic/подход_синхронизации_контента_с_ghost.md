# Подход к синхронизации контента сайта с Ghost CMS

Документ фиксирует выбранный подход: локальные markdown-файлы, Ghost Admin API, Git как единый источник истины, двусторонняя синхронизация и коллаборация.

---

## 1. Общая архитектура

```
Локальные файлы: /home/vitaly/31_NewSite
    ↕ скрипты ghost_sync (push / pull)
Ghost CMS (сайт)

    +

Git (GitHub) — версионирование и коллаборация
```

- **Путь к контенту:** `/home/vitaly/31_NewSite` — в **конфигах и скриптах** необходимо указывать именно этот путь (переменная окружения, параметр в `.env` или аргумент скрипта). Скрипты берут отсюда файлы для отправки на Ghost и сюда же сохраняют файлы, загруженные с сайта (pull).
- **Скрипты** работают только с локальными файлами и Ghost API; Git они не вызывают.
- **Git** используется отдельно: коммиты и push — вручную или по расписанию (cron).
- **Одна папка** `/home/vitaly/31_NewSite` — единственный источник контента; две папки (отправленные/скачанные) не используются.

## 2. Структура контента

### 2.1 Папка контента: /home/vitaly/31_NewSite

В конфигурации (например `.env`) и в скриптах задаётся путь к контенту:

- **CONTENT_PATH** (или аналог): `/home/vitaly/31_NewSite`

Все пути к файлам (для push/pull) строятся относительно этой папки.

```
/home/vitaly/31_NewSite/
├── posts/           # посты блога (опционально; можно хранить .md и в корне)
│   ├── example-post.md
│   └── ...
├── pages/           # статические страницы (опционально)
│   ├── about.md
│   └── ...
├── *.md             # markdown-файлы могут лежать и в корне 31_NewSite
└── .sync/           # метаданные синхронизации (опционально)
    ├── sync-state.json
    └── sync.log
```

### 2.2 Метаданные в frontmatter каждого .md

В каждом файле — расширенный YAML frontmatter:

```yaml
---
title: Заголовок поста
slug: example-post
tags: [Тег1, Тег2]
status: published

# Метаданные синхронизации
ghost_id: 5ddc9141c35e7700383b2937
ghost_updated_at: 2026-02-08T19:23:00Z
local_updated_at: 2026-02-08T19:25:00Z
sync_status: synced
last_synced: 2026-02-08T19:25:15Z
---

Содержимое...
```

**Поля синхронизации:**

| Поле | Назначение |
|------|------------|
| `ghost_id` | ID поста/страницы на Ghost (после первой отправки) |
| `ghost_updated_at` | Время последнего изменения на Ghost |
| `local_updated_at` | Время последнего локального изменения |
| `sync_status` | `unsynced` \| `synced` \| `conflict` \| `remote_changed` |
| `last_synced` | Время последней успешной синхронизации |

Опционально: `content_hash` (SHA256) для точного определения изменений по содержимому.

## 3. Операции синхронизации

### 3.1 Push (локально → Ghost)

**Команда:** `python3 scripts/ghost_sync.py push posts/example-post.md` (путь относительно CONTENT_PATH; в конфиге CONTENT_PATH=/home/vitaly/31_NewSite)

**Логика:**

1. Читать файл, парсить frontmatter.
2. Если `ghost_id` пустой — создать новый пост/страницу через Ghost Admin API.
3. Если `ghost_id` есть — обновить существующий (PUT по id).
4. После успешного ответа API:
   - обновить в frontmatter: `ghost_updated_at`, `local_updated_at`, `sync_status: synced`, `last_synced`;
   - при использовании хеша — обновить `content_hash`.

Файл **не удалять** после отправки.

### 3.2 Pull (Ghost → локально)

**Команда:** `python3 scripts/ghost_sync.py pull`

**Логика:**

1. Получить список всех постов/страниц через Ghost Admin API.
2. Для каждой записи:
   - искать локальный файл по `ghost_id` (или по slug);
   - если файла нет — создать новый .md (контент пришёл с Ghost);
   - если файл есть — сравнить `ghost_updated_at` с локальным состоянием:
     - Ghost новее и локально не меняли → перезаписать файл, обновить метаданные;
     - Ghost новее и локально тоже меняли → конфликт (см. п. 4).

### 3.3 Check (какие файлы изменены локально)

**Команда:** `python3 scripts/ghost_sync.py check`

**Логика (варианты):**

- **По mtime:** сравнить время модификации файла (`os.path.getmtime`) с `last_synced` в frontmatter. Если `mtime > last_synced` — файл считаем изменённым.
- **По хешу:** вычислить SHA256 содержимого, сравнить с `content_hash` в frontmatter. Отличие — файл изменён.

Скрипт **не узнает сам**, что был выполнен `git pull`; проверка запускается вручную после `git pull` или через Git hook (post-merge).

## 4. Разрешение конфликтов

Конфликт: одна и та же сущность изменена и локально, и на Ghost.

**Стратегии:**

**A) Last-Writer-Wins (рекомендуется для старта):**

- Сравнить `local_updated_at` и `ghost_updated_at`.
- Побеждает более позднее изменение.
- Проигравшую версию сохранить в файл с суффиксом `.conflict.md` для ручного просмотра.

**B) Ручное разрешение:**

- При конфликте выставить `sync_status: conflict`, не перезаписывать автоматически.
- Команда `ghost_sync.py conflicts` — вывести список конфликтующих файлов.
- Команда `ghost_sync.py resolve <file> --keep local` или `--keep remote` — применить выбранную версию и отправить на Ghost при необходимости.

## 5. Логирование

**Файл:** `/home/vitaly/31_NewSite/.sync/sync.log`

**Формат строки:**  
`timestamp | action | file | status | details`

**Примеры:**

```
2026-02-08 19:25:15 | PUSH   | posts/example-post.md | created | ghost_id: 5ddc9141...
2026-02-08 19:30:22 | PULL   | pages/about.md       | updated | remote changes applied
2026-02-08 19:35:10 | CONFLICT | posts/example-post.md | local=... remote=...
2026-02-08 19:36:45 | RESOLVE | posts/example-post.md | kept: local
```
(пути относительно /home/vitaly/31_NewSite)

Лог используется для аудита, отката и отладки.

## 6. Git и коллаборация

### 6.1 Роль Git

- Git хранит историю контента (markdown + метаданные).
- Коммиты делаются отдельно от вызовов скрипта (вручную или по расписанию).
- Репозиторий может быть private; при необходимости — public для упрощения доступа коллег.

### 6.2 Workflow с коллегой

1. Коллега: `git clone`, правка файла в `/home/vitaly/31_NewSite/` (или в своей копии репозитория с тем же путём к контенту), `git commit`, `git push`.
2. Вы: `git pull` — получаете изменения в локальные файлы.
3. Вы: `python3 scripts/ghost_sync.py check` — смотрите, какие файлы помечены как изменённые.
4. Вы: `python3 scripts/ghost_sync.py push <file>` или `push-all` — отправляете изменения на Ghost.
5. При необходимости: `git commit` и `git push` для фиксации итогового состояния (в т.ч. обновлённых метаданных после sync).

### 6.3 Автоматизация Git (опционально)

- **Cron:** периодически (например, раз в час) выполнять `git add` для папки контента (например `git add /home/vitaly/31_NewSite/` или соответствующую директорию в репозитории), `git commit`, `git push`.
- **Post-merge hook:** после `git pull` вызывать `ghost_sync.py auto-sync` (check + push изменённых файлов).

## 7. Определение изменённых файлов после git pull

Скрипт **не получает события от Git**. Варианты:

1. **Вручную:** после `git pull` запускать `ghost_sync.py check`, затем `push` для нужных файлов.
2. **Git hook:** в `.git/hooks/post-merge` вызвать `ghost_sync.py auto-sync`.
3. **Критерий «изменён»:** по `mtime` файла относительно `last_synced` или по отличию `content_hash` от текущего хеша содержимого.

---

## 8. Итоговые принципы

1. **Одна папка контента** — `/home/vitaly/31_NewSite`; в конфигах и скриптах путь к контенту должен быть указан явно как этот каталог.
2. **Файлы не удаляются** после отправки на Ghost; обновляются только метаданные в frontmatter.
3. **Git** — слой версионирования и коллаборации; синхронизация с Git — отдельный шаг (вручную или cron/hook).
4. **Скрипты** не вызывают Git; они только читают/пишут файлы и работают с Ghost API.
5. **Конфликты:** сначала Last-Writer-Wins + сохранение проигравшей версии в `.conflict.md`; при необходимости — ручное разрешение через `resolve`.
6. **Лог** `.sync/sync.log` ведётся для всех значимых операций sync.
7. **Изменённые файлы** определяются по mtime или content_hash при явном запуске `check` или `auto-sync`.
